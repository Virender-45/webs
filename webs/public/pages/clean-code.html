Clean Code Practices

Writing code that future-you (and your teammates) will actually thank you for — readable, consistent, and built to last.

Philosophy

Code is Read More Than Written

Clean code isn't a luxury — it's respect for everyone who touches the codebase after you. I follow naming conventions, keep functions small and purposeful, and avoid "clever" tricks that obscure intent.

I believe documentation should exist in the code itself: descriptive names, logical structure, and self-evident logic reduce the need for excessive comments.

Principles I Follow
📌

DRY

Don't Repeat Yourself — extract reusable logic.

🎯

KISS

Keep It Simple, Stupid — prefer clarity over cleverness.

🔧

SRP

Single Responsibility — one function, one job.

🧪

Testable

Write code that's easy to verify and debug.

Before & After

Refactoring in Practice

❌ Before — messy, unclear intent:

function x(a, b) { let r = 0; for(let i=0; i<a.length; i++) { if(a[i] > b) r++; } return r; }

✅ After — self-documenting, purposeful:

// Count how many scores exceed the passing threshold function countPassingScores(scores, passingThreshold) { return scores.filter(score => score > passingThreshold).length; }

Habits

Daily Practices

Name things honestly

If a variable is hard to name, the logic might be too complex — simplify first.

Refactor before shipping

Never leave "I'll clean this up later" in the codebase. Later never comes.

Comment the why, not the what

The code says what it does. Comments explain edge cases and business logic.

Iterate, don't over-engineer

Start simple, let complexity emerge from real requirements — not imagined ones.
